# QA フェーズ — 品質確認

reviewer が承認した実装を実際に動作確認し、バグを見つけて修正します。
テスト計画（PLAN.md）に沿った確認を実行し、発見したバグごとにリグレッションテストを生成します。

## Iron Law（鉄則）

**調査なしに修正しない。** 根本原因を特定してから対処する。

症状を見てすぐコードを変えるのではなく、
なぜそのバグが起きているのかを理解してから修正する。
3 回試みて原因が特定できない場合は、ユーザーに報告して判断を仰ぐ。

---

## 判断フロー

```
1. PLAN.md からテスト計画を読み込む
2. 動作確認チェックリストを実行する
3. バグが出たら根本原因を調査する
4. 軽微なバグは自律修正 → リグレッションテスト生成
5. 重大なバグはユーザーに報告してから修正
6. 全項目が完了したら QA レポートを作成する
7. 合格なら SHIP フェーズへ引き渡す
```

---

## Step 1: テスト計画の読み込み

`projects/<プロジェクト名>/PLAN.md` の「テスト計画」セクションを読み込む。

PLAN.md が存在しない場合（計画なしで実装された場合）:
- 変更されたファイルを確認し、QA チェックリストをその場で作成する
- 次回からは THINK / PLAN フェーズを先行させるよう secretary へ連携する

---

## Step 2: 動作確認チェックリストの実行

テスト計画の各項目を実行し、期待する結果と一致するか確認する。

### 確認の優先順位

| 優先度 | 確認対象 |
|-------|---------|
| 1（必須） | 機能のハッピーパス（正常系） |
| 2（必須） | エラー系・エッジケース |
| 3（必須） | リグレッション（既存機能が壊れていないか） |
| 4（推奨） | アクセシビリティ（frontend タスクの場合） |

### SharePoint 環境での確認観点

- スクリプトがページ読み込み後に正しく動作するか（タイミング問題）
- 権限の違うユーザーでの挙動（管理者 / 一般ユーザー）
- モバイルブラウザでの表示崩れがないか
- 既存の SharePoint 標準 UI を壊していないか

### Frontend の確認観点

- デザイン仕様との一致（Figma や PLAN.md の定義通りか）
- レスポンシブ対応（PC / タブレット / モバイル）
- キーボードのみで全操作できるか（Tab・Enter・Escape）
- スクリーンリーダーで意味が伝わるか（`a11y-check` スキル を併用）

---

## Step 3: バグ発見時の対応

### バグの分類と対処ルール

| 分類 | 判断基準 | 対処 |
|------|---------|------|
| **軽微** | 見た目・文言・マージン調整など機能に影響しない | 自律修正 + レポートに記録 |
| **機能バグ** | 機能が動かない・想定と異なる挙動 | ユーザーに報告してから修正 |
| **データ破損リスク** | リストの更新・削除・権限変更など | 必ずユーザーに報告・承認を得てから修正 |
| **再現不能** | 3 回試みて再現しない | 「環境依存の可能性あり」として記録し続行 |

### 根本原因の調査手順（Iron Law の実践）

```
1. バグを再現できるか確認する
   → 再現できない場合は「再現不能」として記録して次へ

2. バグが発生する条件を特定する
   → どんな操作で、どんな状態のときに起きるか

3. コードの該当箇所を特定する
   → エラーメッセージ・コンソールログ・差分から原因を追う

4. 修正方針を決める
   → 副作用がないか確認してから修正する

5. 修正後に再確認する
   → バグが解消されたか、新たなバグが生まれていないか
```

### リグレッションテストの生成

バグを修正した場合、同じバグが再発しないようにテスト記録を残す。

PLAN.md の「テスト計画」に追記する形式:

```markdown
### リグレッションテスト（QA フェーズで追加）
- [ ] [バグの概要]: [確認方法と期待する結果]
  - 発見日: YYYY-MM-DD
  - 原因: [根本原因の概要]
  - 修正コミット: [コミットメッセージ or SHA]
```

---

## Step 4: ユーザーへの報告が必要なケース

以下のいずれかに該当する場合は、修正前にユーザーに報告する:

- 機能が完全に動かない（設計の見直しが必要な可能性）
- PLAN.md の仕様と実装が大きく食い違っている
- 修正に伴い他のファイルへの広範な変更が必要になる
- セキュリティ上の問題が見つかった（権限バイパス・XSS 等）
- 3 回試みても原因が特定できないバグがある

報告フォーマット:

```
## QA 中断 — 確認が必要です

**プロジェクト**: [プロジェクト名]
**発見した問題**: [バグの概要]
**影響範囲**: [どの機能・ページに影響するか]
**現在の状況**: [どこまで調査したか]

### 判断をお願いしたいこと
A) [修正案 A — 概要と影響]
B) [修正案 B — 概要と影響]
C) 設計を見直す（PLAN フェーズに戻る）
```

---

## Step 5: QA レポートの作成

全確認項目が完了したら QA レポートを作成する。

PLAN.md の末尾に追記する:

```markdown
---
## QA レポート

**QA 実施日**: YYYY-MM-DD
**担当**: frontend
**結果**: 合格 ✅ / 条件付き合格 ⚠️ / 不合格 ❌

### 確認結果サマリー
| 項目 | 結果 | 備考 |
|------|------|------|
| ハッピーパス | ✅ / ❌ | |
| エラー系 | ✅ / ❌ | |
| リグレッション | ✅ / ❌ | |
| アクセシビリティ | ✅ / ❌ / N/A | |

### 発見・修正したバグ
| バグ概要 | 重要度 | 対処 |
|---------|-------|------|
| [バグ] | 軽微 / 機能 | 修正済み / 報告済み |

### 再現不能・保留
- [あれば記載]

### SHIP フェーズへの申し送り事項
- [次の担当者（reviewer）が知っておくべきこと]
```

---

## Step 6: SHIP フェーズへの引き渡し

QA レポートで「合格」が出たら secretary へ報告し、reviewer の `merge-pr` スキルへ引き渡す。

```
## QA フェーズ完了 — SHIP フェーズへ

**プロジェクト**: [プロジェクト名]
**PR**: #[番号]
**QA 結果**: 合格 ✅

修正コミット数: [N 件]
リグレッションテスト追加: [N 件]

SHIP フェーズへ進んでください。（reviewer: merge-pr → secretary: ドキュメント更新）
```
