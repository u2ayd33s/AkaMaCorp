# PLAN フェーズ — 実装計画承認

コードを書く前にアーキテクチャ・依存関係・エッジケース・テスト計画を確定します。
PLAN.md を設計書として完成させ、ユーザーの承認を経てから BUILD フェーズへ進みます。

## 判断フロー

```
1. PLAN.md を読み込む（THINK フェーズの成果物）
2. 技術的実現可能性を検証する
3. アーキテクチャ・ファイル構成を設計する
4. エッジケース・失敗パターンを洗い出す
5. テスト計画を作成する
6. PLAN.md に追記してユーザーの承認を求める
7. 承認されたら BUILD フェーズへ引き渡す
```

**鉄則: ユーザーの承認なしに BUILD フェーズへ進まない。**

---

## Step 1: PLAN.md の読み込み

`projects/<プロジェクト名>/PLAN.md` を読み込む。
THINK フェーズが完了していない場合は `plan-think` スキルを先に実行する。

確認すべきポイント:
- 課題・背景が明確か
- 採用アプローチが決まっているか
- MVS（最小有効実装）の範囲が定義されているか

---

## Step 2: 技術的実現可能性の検証

採用アプローチに対して以下を確認する:

### SharePoint / M365 系タスクの場合
- SharePoint のバージョン（Online か Server か）に依存しないか
- 必要な権限（サイト管理者・テナント管理者）が実行時に確保できるか
- Graph API / REST API の呼び出し制限（スロットリング）に引っかからないか
- SPFx を使う場合、ターゲットの Node.js / SPFx バージョンと互換性があるか

### Frontend 系タスクの場合
- 既存のデザインシステム・コンポーネントと整合するか
- 対応ブラウザ・デバイスの範囲は明確か
- 既存ページへの影響範囲はコントロールできるか

### 共通
- 既存スキル・コンポーネントで再利用できるものはないか
- 外部依存（ライブラリ・API）を新たに増やす場合、それは本当に必要か

---

## Step 3: アーキテクチャ設計

PLAN.md の「アーキテクチャ設計」セクションに以下を記入する:

```markdown
## アーキテクチャ設計

### 変更対象ファイル
| ファイル | 変更種別 | 概要 |
|---------|---------|------|
| [ファイルパス] | 新規 / 更新 / 削除 | [何をするか] |

### データフロー
[入力 → 処理 → 出力 の流れを記述。複雑な場合は図を ASCII で描く]

### 依存関係
- 既存コンポーネント: [使い回すもの]
- 新規作成: [作るもの]
- 外部依存: [ライブラリ・API]

### 実装順序
1. [最初にやること]
2. [次にやること]
3. ...
```

**コミット分割の指針（gstack 準拠）:**
実装が複数の論理的変更を含む場合、コミットを分割する指針も記録する。
例: 「リネームと機能追加は別コミット」「テストと実装は別コミット」

---

## Step 4: エッジケース・失敗パターンの洗い出し

以下の観点で「壊れるパターン」を先に列挙する:

| カテゴリ | 確認観点 |
|---------|---------|
| **入力異常** | 空文字・null・想定外の型・長すぎる値 |
| **権限・認証** | 権限のないユーザーが操作した場合 |
| **ネットワーク** | API 失敗・タイムアウト・スロットリング |
| **既存データ** | すでにデータが存在する場合 / 存在しない場合 |
| **並行操作** | 複数ユーザーが同時に操作した場合 |
| **SharePoint固有** | ページが読み込まれる前にスクリプトが実行される場合 |

PLAN.md への記入形式:

```markdown
## エッジケース・失敗パターン

| ケース | 想定される影響 | 対処方針 |
|-------|--------------|---------|
| [状況] | [何が起きるか] | [どう対処するか] |
```

---

## Step 5: テスト計画の作成

BUILD フェーズと QA フェーズで使うテスト計画を作成する。

```markdown
## テスト計画

### 動作確認チェックリスト（QA フェーズで使用）
- [ ] [確認項目1]: [期待する結果]
- [ ] [確認項目2]: [期待する結果]
- [ ] エラー系: [エラー時の挙動確認]

### アクセシビリティ確認（frontend 担当の場合）
- [ ] キーボードのみで操作できるか
- [ ] スクリーンリーダーで意味が伝わるか
- [ ] 色コントラストが WCAG 2.1 AA を満たしているか

### リグレッション確認
- [ ] 既存機能 [機能名] が壊れていないか
- [ ] 既存ページ [ページ名] の表示が崩れていないか
```

---

## Step 6: ユーザーへの承認依頼

PLAN.md への追記が完了したら、以下の形式でユーザーに報告・承認を求める:

```
## PLAN フェーズ — 承認依頼

**プロジェクト**: [プロジェクト名]
**担当エージェント**: [sharepoint / frontend / 両方]

### 設計のポイント
- [アーキテクチャで重要な判断を 2〜3 点]

### 主なエッジケース対処
- [対処が必要なリスクを 2〜3 点]

### テスト項目数
- 動作確認: N 項目
- リグレッション: N 項目

---
**この計画で BUILD フェーズへ進んでよいですか？**
修正が必要な場合はお知らせください。
```

---

## Step 7: BUILD フェーズへの引き渡し

承認を受けたら `route-agent` スキルを使い、担当エージェントへ割り振る。

引き渡し時に伝えること:
1. PLAN.md のパス（`projects/<プロジェクト名>/PLAN.md`）
2. 実装順序（Step 3 で定義したもの）
3. 特に注意すべきエッジケース

PLAN.md のステータスを更新する:

```markdown
## ステータス
- フェーズ: PLAN 承認済み / BUILD 進行中
- 承認日: YYYY-MM-DD
```
