# THINK フェーズ — 企画立案

コードを書く前に「本当に作るべきものは何か」を問い直します。
ユーザーの要求の背景にある課題を掘り下げ、最小有効実装（MVS）を特定して PLAN.md に記録します。

## 判断フロー

```
1. ユーザーの要求を聞く
2. 背景・ペインを掘り下げる（5つの問い）
3. 前提を崩す問いを投げかける
4. 実装アプローチを 2〜3 案提示する
5. ユーザーと合意し、推奨案を決定する
6. PLAN.md を作成して PLAN フェーズへ引き渡す
```

---

## Step 1: 背景・ペインの掘り下げ

ユーザーの要求を受けたら、以下の観点で確認する。
全部聞く必要はない。要求の不明確な部分に絞って質問する。

| 問い | 目的 |
|------|------|
| 「なぜそれが必要ですか？」 | 機能ではなく課題を特定する |
| 「今はどうやって対処していますか？」 | 現状の回避策を把握する |
| 「誰が使いますか？」 | ユーザー像を明確にする |
| 「どんな状況で困りますか？具体例は？」 | 抽象的な要求を具体化する |
| 「これができたら何が変わりますか？」 | 真のゴールを確認する |

**注意:** 質問攻めにしない。1〜2 問に絞り、会話で掘り下げる。

---

## Step 2: 前提を崩す問い

要求をそのまま受け入れず、以下の視点で問い直す。

- 「本当にこの機能が必要か、それとも別の解決策があるか？」
- 「今すぐ作る必要があるか、後回しにできるか？」
- 「全部作る必要があるか、最小限で済む部分はどこか？」
- 「既存の機能・スキルで代用できないか？」

ユーザーが同意した場合は要求を修正する。
ユーザーが反論した場合は理由を確認し、要求の根拠を深める。

---

## Step 3: 実装アプローチの提案

2〜3 案を提示する。必ず以下を含める:

| 項目 | 内容 |
|------|------|
| アプローチ名 | 短い名前（例: 「シンプル案」「完全実装案」） |
| 概要 | 何をどう作るか（1〜2 文） |
| メリット | なぜこの案が良いか |
| デメリット/リスク | 何が犠牲になるか、何が難しいか |
| 担当エージェント（予定） | sharepoint / frontend / 両方 |
| 規模感 | 小 / 中 / 大 |

**Boil the Lake 原則:**
AIが実装する場合、「少し手を抜いた案」と「完全な実装」のコスト差は小さい。
「完全実装で十分に実現可能（Lake）」な場合は完全実装を推奨する。
「システム全体に影響する大規模改修（Ocean）」は範囲外と明示する。

---

## Step 4: ユーザーとの合意

提案後、以下を確認してから PLAN.md を作成する:

1. どのアプローチを採用するか
2. 最小有効実装（MVS）の範囲 ― 「まず何を動かすか」
3. 優先度 ― 今すぐ着手 / 後回し / 保留

**ユーザー承認なしに PLAN フェーズへ進まない。**

---

## Step 5: PLAN.md の作成

`projects/<プロジェクト名>/PLAN.md` を以下の形式で作成する。
既存のプロジェクトの場合は PLAN.md を上書き更新する。

```markdown
# [機能名 / 改善名]

## ステータス
- フェーズ: THINK 完了 / PLAN 承認待ち
- 作成日: YYYY-MM-DD
- 担当エージェント（予定）: [sharepoint / frontend / 両方]

## 課題・背景
[ユーザーが本当に困っていること。機能の説明ではなくペインを書く]

## ユーザーが求めているもの
[要求の背後にある真のゴール]

## 採用するアプローチ
[選んだアプローチの概要]

**採用理由:**
[なぜこのアプローチを選んだか]

## 見送ったアプローチ
- [案名]: [見送り理由]

## 最小有効実装（MVS）
[まず動かす最小限のもの。ここまでが Phase 1]

## 今回のスコープ外
[将来的にやるかもしれないが今回はやらないこと]

---
<!-- 以下は PLAN フェーズで secretary が記入する -->
## アーキテクチャ設計
（PLAN フェーズで記入）

## テスト計画
（PLAN フェーズで記入）
```

---

## ユーザーへの報告フォーマット

PLAN.md 作成後、以下の形式で報告する:

```
## THINK フェーズ完了

**要求の本質**: [ユーザーが本当に求めているもの]

**採用アプローチ**: [アプローチ名]
> [概要 1〜2 文]

**最小有効実装（MVS）**: [まず動かすもの]

**次のステップ**: PLAN フェーズへ進んでよいですか？
アーキテクチャ設計とテスト計画を確定します。（`plan-review` スキルを使用）
```

---

## スキップしてよい場合

以下のタスクは THINK フェーズを省略して BUILD から開始できる:
- バグ修正（原因が明確で 1 ファイル以内の変更）
- 既存スキルへの小規模追記（例: チェック項目を 1 つ追加）
- ドキュメントのみの変更

スキップ判断は secretary が行い、ユーザーに「THINK フェーズをスキップして BUILD から始めます」と伝える。
