# AkaMaCorp スプリントフロー

gstack の思想（Think → Plan → Build → Review → QA → Ship → Reflect）を
MyCorp の 4 エージェント構成に適用したチーム運用プロセス。

---

## フロー全体図

```
ユーザーの要求
      ↓
┌─────────────────────────────────────────────────────────┐
│  Phase 1: THINK（企画立案）                              │
│  担当: secretary / スキル: plan-think                    │
│  出力: PLAN.md（設計ドキュメント）                        │
└─────────────────┬───────────────────────────────────────┘
                  ↓ PLAN.md を渡す
┌─────────────────────────────────────────────────────────┐
│  Phase 2: PLAN（実装計画承認）                           │
│  担当: secretary / スキル: plan-review                   │
│  出力: 承認済み実装計画（アーキテクチャ・テスト計画含む） │
└─────────────────┬───────────────────────────────────────┘
                  ↓ 実装計画を渡す
┌─────────────────────────────────────────────────────────┐
│  Phase 3: BUILD（実装）                                  │
│  担当: designer / sharepoint / frontend（タスクに応じて選択）│
│  出力: feature ブランチの PR                             │
└─────────────────┬───────────────────────────────────────┘
                  ↓ PR を渡す
┌─────────────────────────────────────────────────────────┐
│  Phase 4: REVIEW（コードレビュー）                       │
│  担当: reviewer / スキル: review-pr                      │
│  出力: レビュー結果（承認 or 差し戻し）                   │
└─────────────────┬───────────────────────────────────────┘
                  ↓ 承認済み PR を渡す
┌─────────────────────────────────────────────────────────┐
│  Phase 5: QA（品質確認）                                 │
│  担当: frontend / スキル: qa-check                       │
│  出力: QA レポート + リグレッションテスト                 │
└─────────────────┬───────────────────────────────────────┘
                  ↓ QA 承認済みを渡す
┌─────────────────────────────────────────────────────────┐
│  Phase 6: SHIP（リリース）                               │
│  担当: reviewer (merge-pr) → secretary (doc 更新)        │
│  出力: マージ済み main + ドキュメント更新済み             │
└─────────────────┬───────────────────────────────────────┘
                  ↓ 完了したスプリントを記録
┌─────────────────────────────────────────────────────────┐
│  Phase 7: REFLECT（振り返り）                            │
│  担当: secretary / スキル: nfd-crystallize               │
│  出力: 振り返りログ + 知識結晶化                          │
└─────────────────────────────────────────────────────────┘
```

## 各フェーズ詳細

### Phase 1: THINK（企画立案）

| 項目 | 内容 |
|------|------|
| 担当 | secretary |
| スキル | `plan-think`（新設） |
| トリガー | ユーザーが新機能・改善・相談を持ち込んだとき |
| 目的 | 「何を作るか」ではなく「何を作るべきか」を問い直す |

**やること:**
1. ユーザーの要求の背景にある課題・ペインを掘り下げる
2. 前提を崩す問いを投げかける（「本当にそれが必要か？」）
3. 複数の実装アプローチを提案（最低 2 案）
4. 最小有効実装（MVS）を特定する

**出力物（PLAN.md）:**
```
## 課題・背景
## ユーザーが本当に求めているもの
## 実装アプローチ案（2〜3案）
## 推奨アプローチと理由
## 最小有効実装（MVS）の定義
## 担当エージェント（予定）
```

**スキップしてよい場合:**
- 軽微なバグ修正（1 ファイル以内）
- 既存スキルの小修正（ドキュメント変更のみ）

---

### Phase 2: PLAN（実装計画承認）

| 項目 | 内容 |
|------|------|
| 担当 | secretary |
| スキル | `plan-review`（新設） |
| トリガー | PLAN.md が完成し、ユーザーが承認したとき |
| 目的 | アーキテクチャ・依存関係・テスト計画を確定しコード実装前にリスクを潰す |

**やること:**
1. PLAN.md を読み込み、技術的実現可能性を検証する
2. ファイル構成・依存関係・インターフェース設計を定義する
3. エッジケース・失敗パターンを列挙する
4. テスト計画（何をどうテストするか）を作成する
5. ユーザーに承認を求めてから BUILD フェーズへ進む

**出力物（PLAN.md に追記）:**
```
## アーキテクチャ設計
## ファイル構成（変更対象）
## エッジケース・失敗パターン
## テスト計画
## 実装担当エージェントの確定
```

**重要:** ユーザーの承認なしに BUILD フェーズへ進まない。

---

### Phase 3: BUILD（実装）

| 項目 | 内容 |
|------|------|
| 担当 | designer / sharepoint / frontend（タスクによって選択） |
| スキル | 各専門スキル（wireframe, figma-design, spfx-component, ui-component 等） |
| トリガー | 実装計画がユーザーに承認されたとき |
| 目的 | 計画通りに実装し、PR を作成する |

**やること:**
1. PLAN.md の実装計画を読み込む
2. feature ブランチを作成して実装する
3. テスト計画に沿ったテスト/確認項目を実施する
4. PR を作成し、REVIEW フェーズへ引き渡す

**コミットルール（gstack 準拠）:**
- 1 コミット = 1 つの論理的変更
- リネーム・リファクタ・新機能・テストは別々のコミットに
- コミットメッセージは日本語で「何をなぜ変えたか」を記述

**出力物:** feature ブランチ + PR（説明に PLAN.md 参照を記載）

---

### Phase 4: REVIEW（コードレビュー）

| 項目 | 内容 |
|------|------|
| 担当 | reviewer |
| スキル | `review-pr` |
| トリガー | PR が作成されたとき |
| 目的 | CI を通るが本番で壊れる問題・セキュリティ・品質を確認する |

**チェック観点（スマートルーティング）:**

| PR の種類 | 追加チェック |
|----------|------------|
| SharePoint / SPFx | 権限設定・API スロットリング・テナント依存 |
| Frontend / UI | デザイン整合性・アクセシビリティ基準 |
| ドキュメントのみ | 自動承認可（内容確認のみ） |
| 設定ファイル変更 | 必ずユーザーに報告してから判断 |

**出力物:** 承認コメント or 差し戻しコメント（具体的な修正指示付き）

---

### Phase 5: QA（品質確認）

| 項目 | 内容 |
|------|------|
| 担当 | frontend |
| スキル | `qa-check`（新設） |
| トリガー | reviewer が承認したとき |
| 目的 | 実際の動作確認・アクセシビリティ・リグレッション確認 |

**やること:**
1. 実装計画のテスト計画を読み込む
2. 動作確認チェックリストを実行する
3. アクセシビリティ（WCAG 2.1 AA）を確認する（`a11y-check` 併用）
4. バグを発見した場合は修正 + リグレッションテストを生成する
5. QA レポートを作成する

**Iron Law（鉄則）:** 調査なしに修正しない。根本原因を特定してから対処する。

**バグ発見時のルール:**
- 軽微（見た目・文言）→ 自律修正 + レポートに記録
- 機能バグ → ユーザーに報告してから修正判断
- 3 回試みて再現不能 → 「環境依存の可能性あり」として記録

**出力物:** QA レポート（合格 or バグ一覧 + 修正コミット）

---

### Phase 6: SHIP（リリース）

| 項目 | 内容 |
|------|------|
| 担当 | reviewer → secretary |
| スキル | `merge-pr` → ドキュメント更新 |
| トリガー | QA レポートで合格が出たとき |
| 目的 | main へのマージ + ドキュメントを最新化する |

**手順:**
1. reviewer: main との差分確認 → マージ実行
2. secretary: 変更内容に照らして以下を自動更新:
   - README.md（機能変更があれば）
   - SKILL.md（スキル変更があれば）
   - agent-registry.md（エージェント変更があれば）
   - CHANGELOG.md（存在すれば）

**ドキュメント更新の原則:**
- ユーザー視点で書く（「実装詳細」ではなく「何ができるようになったか」）
- 古い記述は削除する（後方互換コメントを残さない）

---

### Phase 7: REFLECT（振り返り）

| 項目 | 内容 |
|------|------|
| 担当 | secretary |
| スキル | `nfd-crystallize` |
| トリガー | SHIP 完了後 / 月次（毎月 16 日）/ 日次ログ 20 件超過時 |
| 目的 | 暗黙知を結晶化し、次のスプリントの品質を上げる |

**やること:**
1. 完了したスプリントを振り返り、以下を記録する:
   - うまくいったこと（再現したい）
   - 失敗・詰まったこと（次回避けたい）
   - 発見したパターン
2. memory/ の insights.md / error-patterns.md を更新する
3. 必要に応じてスキルを改善する（CC サイクル）

## フェーズのスキップルール

小さなタスクは不要なフェーズをスキップできる。判断基準:

| タスク規模 | 必須フェーズ |
|----------|------------|
| 小（1 ファイル・バグ修正） | BUILD → REVIEW → SHIP |
| 中（新機能・複数ファイル） | THINK → PLAN → BUILD → REVIEW → QA → SHIP |
| 大（新エージェント・設計変更） | 全フェーズ + ユーザー承認チェックポイント |

## discrepancy-check のスキル・メモリ記録フロー

差分チェックエージェントが作業で得た知識を蓄積・反映する際のルール。

```
discrepancy-check: スキル改善 or メモリ記録
    ↓ feature ブランチ + PR 作成
secretary: PRレビュー
  - スキル定義の整合性・目的の明確さ
  - メモリ記録の適切さ（タグ・内容）
  - 不足・質問があれば差し戻し
    ↓ 不足なし
reviewer: 最終確認 → マージ
  - ファイル構成・フォーマット準拠
  - 差し戻し事項がなければ merge-pr
    ↓ マージ
secretary: REFLECT（必要に応じて知識結晶化）
```

**対象ファイル:**
- スキル更新: `agents/06_discrepancy-check/skills/<スキル名>/SKILL.md`
- メモリ記録: `memory/daily/YYYY-MM-DD.md`（`[INSIGHT]` / `[ERROR]` / `[PATTERN]`）

## 品質ゲートのルール

各フェーズの完了条件（次のフェーズへ進むための条件）:

| フェーズ | 完了条件 |
|---------|---------|
| THINK | PLAN.md が作成され、ユーザーが内容を確認した |
| PLAN | ユーザーが実装計画を承認した |
| BUILD | PR が作成され、自己チェックを完了した |
| REVIEW | reviewer が承認コメントを付けた |
| QA | QA レポートで合格が出た（またはバグが全て修正された） |
| SHIP | main へのマージが完了し、ドキュメントが更新された |
| REFLECT | memory/ が更新された |

## 全エージェント共通ルール

すべてのエージェント・すべての作業に適用される運用ルール。個別の CLAUDE.md には記載しない（ここが唯一の定義場所）。

### タイムアウト・中断ルール

| 状況 | 対応 |
|------|------|
| ツール呼び出し・処理が **10分以上** 応答しない | 作業を中断し、改善案をユーザーに提示する |
| 中断後の対応手順 | ① 何が止まっているかを特定 → ② 原因を分析 → ③ 改善案を提示（リトライ・代替手段・スキップ等） |
| やってはいけないこと | 同じ操作をリトライし続ける・黙って待ち続ける |

**適用範囲:** 全リポジトリ・全エージェント・全フェーズ
